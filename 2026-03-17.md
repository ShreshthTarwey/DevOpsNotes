# Docker Layers and Image Registries - March 17, 2026

This document provides comprehensive notes on Docker image layers, their advantages, and various image registry types used in Docker ecosystems.

## 1. Layers in Docker Images

### What are Layers?

Layers are the fundamental building blocks of Docker images. Each layer represents a set of file system changes (additions, deletions, or modifications) that are applied on top of the previous layer. These layers are stacked together to form a complete Docker image.

### How Layers Work

- **Stackable Architecture:** Each instruction in a Dockerfile creates a new layer
- **Immutable:** Once created, layers cannot be modified
- **Read-only:** All layers except the top (container layer) are read-only
- **Union File System:** Layers are combined using union mount technology to appear as a single file system

### Layer Structure

```
┌─────────────────────────────────┐
│  Container Layer (Read-write)   │  <- Running container adds this
├─────────────────────────────────┤
│  Layer N (FROM, RUN, COPY, etc) │  <- Latest image layer
├─────────────────────────────────┤
│  Layer N-1                      │
├─────────────────────────────────┤
│  Layer N-2                      │
├─────────────────────────────────┤
│  ...                            │
├─────────────────────────────────┤
│  Layer 2                        │
├─────────────────────────────────┤
│  Layer 1 (Base Image)           │  <- Bottom-most layer
└─────────────────────────────────┘
```

### Dockerfile Instructions and Layers

Each instruction in a Dockerfile creates a new layer:

| Instruction | Creates Layer? | Purpose |
|-----------|----------------|---------|
| `FROM` | Yes | Base image layer |
| `RUN` | Yes | Execute command and capture changes |
| `COPY` | Yes | Copy files from host to image |
| `ADD` | Yes | Add files (with URL/tar support) |
| `WORKDIR` | No | Set working directory (metadata) |
| `ENV` | No | Set environment variables (metadata) |
| `EXPOSE` | No | Document exposed ports (metadata) |
| `CMD` | No | Default command (metadata) |
| `ENTRYPOINT` | No | Container entry point (metadata) |
| `LABEL` | No | Add metadata labels |
| `USER` | No | Set user for RUN commands |

### Example Dockerfile and Layers

```dockerfile
FROM ubuntu:20.04                      # Layer 1: Base image
RUN apt-get update                     # Layer 2: Install updates
RUN apt-get install -y python3         # Layer 3: Install Python
COPY app.py /app/                      # Layer 4: Copy application
WORKDIR /app                           # Not a layer (metadata only)
CMD ["python3", "app.py"]              # Not a layer (metadata only)
```

This Dockerfile creates 4 layers from the instructions that modify the file system.

## 2. Advantages of Layers

### 2.1 Storage Efficiency (Deduplication)

- **Shared Base Layers:** Multiple images can share the same base layers
- **Reduced Storage:** Only one copy of each layer is stored on disk
- **Example:** Both `ubuntu:20.04` and `debian:11` based images share common system library layers if built on similar base images

```
Image A (100MB total)
├── Base Layer (50MB) - shared
├── Layer 2 (30MB) - unique
└── Layer 3 (20MB) - unique

Image B (110MB total)
├── Base Layer (50MB) - shared (no duplicate)
├── Layer 2 (30MB) - unique
└── Layer 3 (30MB) - unique

Total Storage Used: 50 + 30 + 20 + 30 + 30 = 160MB (not 210MB)
```

### 2.2 Faster Image Building

- **Caching:** Docker caches layers during build process
- **Reuse:** Unchanged layers are reused from cache
- **Speed:** Subsequent builds are much faster
- **Incremental Building:** Only modified layers need to be rebuilt

```
Build 1: RUN apt-get install -y python3  (takes 45 seconds)
Build 2: RUN apt-get install -y python3  (uses cache, instant)
```

### 2.3 Faster Image Pulling

- **Partial Downloads:** Only missing layers need to be downloaded
- **Resume Capability:** Interrupted downloads can resume
- **Bandwidth Savings:** Significantly reduces data transfer
- **Parallel Downloads:** Multiple layers can be downloaded simultaneously

### 2.4 Efficient Image Distribution

- **Delta Updates:** Only changed layers need distribution
- **Reduced Network Traffic:** Dramatically lower bandwidth requirements
- **Selective Caching:** Users can selectively cache needed layers
- **Content Addressable:** Layers identified by SHA256 hash ensure authenticity

### 2.5 Security Benefits

- **Isolate Changes:** Security updates can be deployed as separate layers
- **Audit Trail:** Layer history provides clear change documentation
- **Rollback Capability:** Easy to revert to previous layer versions
- **Vulnerability Scanning:** Each layer can be scanned independently

### 2.6 Development Productivity

- **Faster Iteration:** Only rebuild changed layers
- **Debugging:** Inspect individual layers with `docker history`
- **Customization:** Extend base images with custom layers
- **Modular Approach:** Build reusable base images

### 2.7 Container Runtime Benefits

- **Quick Container Creation:** Layers are read-only, container layer is minimal
- **Copy-on-Write:** Only modified files are copied when container writes
- **Memory Efficient:** Shared read-only layers reduce memory footprint
- **Performance:** Near-instant container startup

## 3. Types of Image Registries

### 3.1 Public Registries

#### Docker Hub

**Description:**
- Official Docker registry managed by Docker Inc.
- Default registry when no host is specified
- Largest repository of public Docker images

**Characteristics:**
- Free public repositories
- Paid private repositories
- Official images (maintained by Docker Inc.)
- Community-contributed images
- Automated builds from GitHub/GitLab

**URL:** `docker.io` (default)

**Usage:**
```bash
docker pull ubuntu                    # Pulls from Docker Hub (default)
docker pull docker.io/library/ubuntu  # Full registry path
```

**Advantages:**
- Large community and image collection
- Easy collaboration
- Official images available
- Free tier generous

**Disadvantages:**
- Dependent on Docker Inc.'s infrastructure
- Rate limiting on free accounts
- Limited security control for public images

---

#### Quay.io

**Description:**
- Container registry operated by Red Hat / CoreOS
- Focus on security and reliability
- Alternative to Docker Hub

**Characteristics:**
- Public and private repositories
- Advanced security scanning
- Automated builds
- Image signing support
- Granular access control

**URL:** `quay.io`

**Usage:**
```bash
docker pull quay.io/coreos/etcd
```

**Advantages:**
- Strong security features
- Reliability and uptime
- Enterprise-grade infrastructure
- Image vulnerability scanning

**Disadvantages:**
- Smaller community than Docker Hub
- Premium features for advanced users

---

#### GitHub Container Registry (GHCR)

**Description:**
- Container registry by GitHub
- Integrated with GitHub repositories
- Native GitHub Actions support

**Characteristics:**
- Free for public images
- Integrated with GitHub workflows
- Same authentication as GitHub
- Support for GitHub Packages

**URL:** `ghcr.io`

**Usage:**
```bash
docker pull ghcr.io/username/image:tag
```

**Advantages:**
- Seamless GitHub integration
- Free tier
- GitHub Actions automation
- One authentication method

**Disadvantages:**
- Smaller ecosystem than Docker Hub
- Limited discovery/search capabilities

---

#### Google Container Registry (GCR)

**Description:**
- Container registry by Google Cloud
- Part of Google Cloud Platform ecosystem
- High-performance, scalable registry

**Characteristics:**
- Integrated with Google Cloud services
- High availability and redundancy
- Vulnerability scanning
- Integration with Google Cloud Build

**URLs:** 
- `gcr.io` (US multi-region)
- `us.gcr.io`, `eu.gcr.io`, `asia.gcr.io` (regional)

**Usage:**
```bash
docker pull gcr.io/project-id/image:tag
```

**Advantages:**
- Enterprise-grade infrastructure
- Google Cloud integration
- Automatic vulnerability scanning
- High performance

**Disadvantages:**
- Requires Google Cloud account
- Pricing based on storage and bandwidth

---

#### Amazon Elastic Container Registry (ECR)

**Description:**
- Container registry by Amazon AWS
- Integrated with AWS services
- Managed container registry service

**Characteristics:**
- Private by default
- AWS IAM integration
- Image lifecycle policies
- Image scanning
- Cross-region replication

**URL Format:** `ACCOUNT_ID.dkr.ecr.REGION.amazonaws.com/REPOSITORY`

**Usage:**
```bash
aws ecr get-login-password --region us-east-1 | docker login --username AWS --password-stdin 123456789012.dkr.ecr.us-east-1.amazonaws.com
docker pull 123456789012.dkr.ecr.us-east-1.amazonaws.com/my-repo:latest
```

**Advantages:**
- AWS ecosystem integration
- Private by default (secure)
- IAM-based access control
- Automatic image scanning

**Disadvantages:**
- AWS account required
- Usage-based pricing
- Regional considerations

---

### 3.2 Private Registries

#### Docker Registry (Self-hosted)

**Description:**
- Official Docker Registry implementation
- Can be self-hosted on any server
- Lightweight and simple

**Characteristics:**
- Open-source software
- Minimal dependencies
- RESTful API
- Storage-flexible (filesystem, S3, etc.)

**Deployment:**
```bash
docker run -d -p 5000:5000 --name registry registry:2
docker tag myimage localhost:5000/myimage
docker push localhost:5000/myimage
```

**Advantages:**
- Full control over infrastructure
- Cost-effective
- No external dependencies
- Simple setup

**Disadvantages:**
- Self-maintenance required
- No built-in UI
- Scaling challenges
- Limited security features (need add-ons)

---

#### Harbor

**Description:**
- Open-source container registry by VMware/CNCF
- Enterprise-grade private registry
- Cloud-native design

**Characteristics:**
- Web UI for management
- Role-based access control (RBAC)
- Image scanning and vulnerability detection
- Image signing and verification
- Replication between registries
- Helm chart repository

**Features:**
- Project-based organization
- Audit logging
- Garbage collection
- Helm integration

**Advantages:**
- Enterprise features
- Full-featured web interface
- Strong security model
- Community support

**Disadvantages:**
- More complex to deploy
- Resource intensive
- Steeper learning curve

---

#### Artifactory (JFrog)

**Description:**
- Universal repository manager by JFrog
- Supports multiple package types
- Enterprise-grade solution

**Characteristics:**
- Container, artifact, and package management
- High availability
- Edge distribution with Artifactory Edge
- Advanced security features
- Integration with CI/CD tools

**Supported Formats:**
- Docker images
- Maven, npm, PyPI, NuGet
- Generic artifacts
- Kubernetes Helm charts

**Advantages:**
- Multi-format support
- Enterprise security
- High performance
- Global distribution

**Disadvantages:**
- Commercial (paid)
- Resource intensive
- Complex configuration

---

#### GitLab Container Registry

**Description:**
- Container registry integrated with GitLab
- Part of GitLab platform
- Developer-friendly

**Characteristics:**
- Project-based repositories
- GitLab CI/CD integration
- User authentication via GitLab
- Container scanning
- Image versioning

**URL Format:** `registry.gitlab.com/namespace/project`

**Advantages:**
- Integrated with GitLab workflow
- GitLab CI/CD automation
- Project isolation
- Built-in permissions

**Disadvantages:**
- Requires GitLab instance
- Smaller community than Docker Hub

## 4. Best Practices for Using Registries

### 4.1 Image Registry Selection

**Use Docker Hub when:**
- Building public, open-source projects
- Need large community and easiest sharing
- Testing and learning Docker

**Use Cloud Registries (GCR, ECR, ACR) when:**
- Already using cloud provider services
- Need seamless cloud integration
- Enterprise security requirements

**Use Private Registries when:**
- Storing proprietary/confidential images
- Need full infrastructure control
- Regulatory/compliance requirements

### 4.2 Image Tagging Strategy

```bash
# Tag format: registry/namespace/repository:tag
docker tag myapp:latest docker.io/username/myapp:latest
docker tag myapp:1.0.0 docker.io/username/myapp:1.0.0

# Push with semantic versioning
docker push docker.io/username/myapp:latest
docker push docker.io/username/myapp:1.0
docker push docker.io/username/myapp:1.0.0
```

### 4.3 Image Layer Optimization

- **Minimize Layers:** Combine related RUN commands
- **Order Matters:** Place frequently changing layers toward end
- **Use .dockerignore:** Exclude unnecessary files
- **Multi-stage Builds:** Reduce final image size

## Summary

### Key Takeaways

1. **Layers Architecture:**
   - Docker images consist of stacked, immutable layers
   - Each Dockerfile instruction creates a new layer
   - Union file system combines layers

2. **Layer Advantages:**
   - Storage efficiency through deduplication
   - Faster builds with caching
   - Reduced image downloads
   - Security and audit benefits
   - Efficient container runtime

3. **Registry Types:**
   - **Public:** Docker Hub, Quay.io, GHCR, GCR
   - **Cloud-managed:** ECR, GCR, ACR
   - **Self-hosted:** Docker Registry, Harbor, Artifactory
   - Each has different strengths and use cases

4. **Selection Criteria:**
   - Public projects → Docker Hub
   - Enterprise → Private registries or cloud registries
   - Cloud-native → Cloud provider registries
   - Multi-format → Artifactory
